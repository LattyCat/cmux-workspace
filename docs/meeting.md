## 第1部: cmuxとは何か ── なぜ今これなのか

**風間**: みんな、今日集まってもらったのは、AIコーディングエージェントの作業環境を根本から見直したいからだ。結論から言う。cmuxを導入して、ワンコマンドで最適なワークスペースを自動構築できる仕組みを作りたい。

**桜井**: あの、cmuxって何ですか？ tmuxの親戚みたいなものですか？

**九条**: 名前は似てるけど、アーキテクチャが根本的に違う。tmuxはサーバー・クライアントモデルのターミナルマルチプレクサで、カーネルのpty層の上に独自のレイヤーを被せている。一方cmuxは、Ghosttyというレンダリングエンジン（libghostty）をベースにしたネイティブmacOSアプリだ。AppKitで動いていて、GPU加速のレンダリングを持っている。

**桜井**: つまり……普通のmacOSアプリなんですね？

**九条**: そう。だからmacOSのウィンドウ管理、フォント描画、IME対応がすべてネイティブに動く。tmuxだとターミナルエミュレータの中にさらに仮想ターミナルを作る二重構造になるが、cmuxにはそのオーバーヘッドがない。そしてここが重要なんだが、cmuxには`cmux.json`という設定ファイルでレイアウトを宣言的に定義できる仕組みがある。さらにソケットAPIとCLIで外部から完全にスクリプト制御できる。

**風間**: そう！ しかもClaude Codeとの統合が最初から設計されてる。`cmux claude-teams`でチームメイトモードが一発で立ち上がるし、OSC 777の通知をサイドバーで拾ってくれる。Claude Codeが入力待ちになったら青いリングで教えてくれるんだ。

**灰原**: 待ってくれ。それは魅力的だが、cmuxはまだ若いプロジェクトだ。READMEを見る限りmacOS 14以上が必須で、セッションリストア時にClaude Codeの状態は復元されない。つまりアプリがクラッシュしたら作業中のClaude Codeセッションは消える。これはtmuxなら`tmux-resurrect`で救えるケースだ。

**九条**: その指摘は正しい。ただし、Claude Codeはそもそもステートレスなセッションだ。会話コンテキストはローカルのメモリ上にあり、tmux-resurrectで復元できるのはシェルの作業ディレクトリとヒストリーだけで、AIとの会話状態は復元されない。Claude Codeの`/resume`機能が実質的なセッション復元手段だ。

**灰原**: ……確かにそうだ。ターミナルマルチプレクサ側のセッション永続化は副次的な要件だな。

**桜井**: なるほど。じゃあcmuxを使う一番のメリットは何なんですか？

**九条**: 3つある。第一に、`cmux.json`による宣言的レイアウト定義。JSON一つでワークスペースの全ペイン構成を記述できる。第二に、CLIとソケットAPIによるスクリプタビリティ。`cmux new-split`、`cmux send`で外部からペインを操作できる。第三に、AIエージェントとのネイティブ統合。通知、サイドバーメタデータ、チームモードが全部組み込まれている。

**風間**: tmuxでこれを再現しようとしたら、tmux + tmuxinator + カスタムスクリプト + 通知デーモンの組み合わせが必要になる。cmuxならワンパッケージだ。

## 第2部: ツール選定 ── 何をペインに載せるか

**風間**: まず要件を整理しよう。必要なのは4つ。ファイルツリー、Markdownプレビュー、gitの変更確認、そしてAIエージェントのメインペイン。プラス補助シェル。

**九条**: ツールを選ぶ前に原則を決めよう。各ツールは次の条件を満たすべきだ。第一に、ターミナル内で完結すること（GUIアプリを別途起動しない）。第二に、インタラクティブTUIであること（watchコマンドでコマンド出力を垂れ流すのではなく）。第三に、起動が高速であること（1秒以上かかるものは論外）。

### ファイルツリー: yaziの検討

**風間**: ファイルツリーはyazi一択だと思う。Rust製で爆速、非同期I/O、画像プレビューまでできる。

**灰原**: rangerやnnn、lfとの比較はしたのか？

**九条**: 実用上、yaziが最適解だ。理由を3つ挙げる。まず起動速度。yaziは非同期設計で、ディレクトリエントリのロードがノンブロッキングだ。巨大なnode_modulesがあるプロジェクトでも初回表示が一瞬で済む。rangerはPython製でコールドスタートが遅い。次にカスタマイズ性。`yazi.toml`の`ratio`パラメータで親ディレクトリ:カレント:プレビューの比率を`[1, 4, 3]`のように指定できる。cmuxの狭いペイン内で使う場合、プレビューペインを無効にして`[0, 1, 0]`にすれば、純粋なファイルリストとして機能する。最後にプラグイン機構。`toggle-pane`プラグインでプレビューの表示/非表示を動的に切り替えられる。

**桜井**: プレビューを無効にする理由は何ですか？

**九条**: cmuxの左ペインは画面全体の25%程度しかない。その中でyaziの3カラムレイアウト（親 + カレント + プレビュー）を出すと、各カラムが狭すぎて何も読めない。カレントディレクトリのリストだけを表示する方が実用的だ。ファイルの中身を見たいときは、AIエージェントに`Read`させればいい。

**桜井**: あ、yaziはファイルを見るためじゃなくて、ファイル構造を把握するためなんですね。

**九条**: その通り。yaziの役割は「地図」だ。中身を読むのはAIエージェントの仕事。

### Git確認: lazygitの検討

**風間**: gitの確認にはlazygitを推したい。ステージング、diff、ログ、ブランチ操作が全部一つのTUIに入ってる。

**灰原**: lazygitはフル機能のgit TUIだ。diffの確認だけなら重すぎないか？ `git diff --stat`をwatchで回す方がシンプルだ。

**九条**: 灰原、その判断基準は間違っている。「シンプル」を追求するあまり、情報密度を犠牲にしている。`watch git diff --stat`は変更ファイルのリストしか出ない。lazygitなら、左パネルでファイルを選択すれば右パネルにインライン差分が出る。さらにステージング、コミット、プッシュまでキーボードだけで完結する。AIエージェントが生成したコードを確認してコミットするワークフローにおいて、lazygitは「確認」と「操作」を一つのペインで完結させる。

**灰原**: それは認める。だが、lazygitはメモリ消費が大きい。巨大リポジトリだと初回のgit statusに時間がかかる。

**九条**: 大規模リポジトリならgitの`fsmonitor`を有効にすればいい。`git config core.fsmonitor true`で、git status自体が高速化される。lazygitはgitの出力をパースしているだけだから、git本体が速くなれば自動的にlazygitも速くなる。

**灰原**: ……わかった。lazygitで行こう。ただし`fsmonitor`の設定はセットアップスクリプトに含めるべきだ。

### Markdownプレビュー: glow vs grip 大論争

**風間**: ここが一番議論が必要だ。選択肢は2つ。ターミナル内で`glow`を使うか、cmuxの内蔵ブラウザペインで`grip`を使うか。

**九条**: まず各ツールの特性を整理しよう。`glow`はCharmbracelet製のCLIマークダウンレンダラだ。ターミナル内でANSIカラーを使って描画する。軽量で高速。ただし重大な制限がある。watchモードが未実装だ。ファイルの変更を検知して自動で再描画する機能がない。

**桜井**: じゃあ、ファイルを保存するたびに手動でglowを再実行するんですか？

**九条**: そうなる。もしくは外部のファイルウォッチャーと組み合わせる。`fswatch`や`watchexec`でファイル変更を検知して`glow`を再実行する仕組みだ。ただし、glowは`watchexec`と組み合わせるとプロセスグループの扱いでバグが出る。具体的には、pagerの出力が表示されなくなり、Ctrl-Cでも終了しない問題が報告されている。

**風間**: そこでgripだ！ gripはGitHub Flavored Markdownをローカルサーバーでレンダリングするツールで、`localhost:6419`でHTMLとして配信する。cmuxの内蔵ブラウザペインで表示すれば、リッチなマークダウンプレビューが手に入る。

**灰原**: gripはGitHub APIを叩くんじゃなかったか？ レートリミットに引っかかるぞ。

**風間**: あ……。

**九条**: 灰原の指摘は半分正しい。gripはデフォルトでGitHub APIを使うが、`--render`オプションを外してローカルレンダリングモードにすれば、APIは不要だ。ただし、その場合のレンダリング品質はGitHub準拠ではなくなる。

**灰原**: もう一つ問題がある。gripはPython製で、`pip install grip`が必要だ。Python環境の依存が増える。さらに、cmuxの内蔵ブラウザペインを使うということは、ペインの種類がターミナルではなくブラウザになる。つまり、同じペインでyaziとマークダウンプレビューをタブ切り替えする場合、一方はターミナルsurface、もう一方はブラウザsurfaceになる。cmuxがこのハイブリッドなsurface構成をサポートしているか確認が必要だ。

**九条**: 鋭い。cmux.jsonのsurface定義を見ると、`type: "browser"`を指定できる。同じペイン内に`type`が異なるsurfacesを混在させることは仕様上可能だ。だが、ここで本質的な問いを立てよう。**マークダウンプレビューに本当にリッチレンダリングが必要か？**

**桜井**: え？ きれいに見えた方がいいに決まってません？

**九条**: そうとは限らない。このワークスペースの目的は何だ？ AIエージェントでコーディングすることだ。マークダウンプレビューの用途は、AIが生成したREADMEやドキュメントの確認、あるいはプランファイルの閲覧だ。そこで求められるのは「構造が正しいか」「見出しレベルが合っているか」「リストが正しくネストされているか」の確認であって、CSSで装飾されたリッチな見た目ではない。

**風間**: うーん……でもせっかくcmuxにブラウザペインがあるなら使いたいじゃないか。

**九条**: 「使えるから使う」は設計ではない。**目的から逆算して選ぶ**のが設計だ。glowのターミナルレンダリングは、ANSI 256色でシンタックスハイライト、見出し、リスト、コードブロックを十分に表現できる。ブラウザペインを使うとその分のメモリとCPUを消費する。WebKitのプロセスが一つ増えるということだ。

**灰原**: 九条に同意する。さらに言えば、ブラウザペインを使うと`grip`サーバープロセスの管理も必要になる。起動、終了、ポート衝突、プロセスの死活監視。シンプルなCLIツール一つで済むならそちらを選ぶべきだ。

**桜井**: でもglowのwatchモードがないんですよね？ リアルタイムで更新されないなら、プレビューの意味が薄くないですか？

**九条**: いい指摘だ。glowのwatchモード問題を解決する方法を提案しよう。`watchexec`との組み合わせにバグがあると言ったが、glowを`-p`（pagerなし）モードで使えばこの問題を回避できる。具体的には：

```bash
watchexec -w . -e md -r -- sh -c 'clear && glow -p README.md'
```

`-p`フラグでpager機能を無効にして標準出力に直接描画させる。`watchexec`が変更検知のたびに`clear && glow -p`を再実行する。これでリアルタイムプレビューが実現する。

**風間**: おお！ それでwatchexecのバグも回避できるのか。

**九条**: pagerを使わないから、プロセスグループの問題が発生しない。ただし、描画のたびに画面がクリアされるので、スクロール位置は保持されない。長いマークダウンの途中を見ていても、更新のたびに先頭に戻る。

**灰原**: 実用上、AIエージェントが編集するマークダウンは常に先頭から確認するケースがほとんどだ。スクロール位置の保持は優先度が低い。

**桜井**: じゃあglow + watchexecの方がいいってことですね？

**九条**: ここでは代替案も提示しておく。`glow`の代わりに`mdcat`というツールもある。Rust製で高速、KittyやiTerm2のインラインイメージプロトコルに対応しているから画像も表示できる。ただし、cmux（Ghosttyベース）がKittyのイメージプロトコルに対応しているかは要確認だ。汎用性と安定性で言えばglowが堅い。

**風間**: よし、glow + watchexecで行こう！

**九条**: もう一歩踏み込もう。監視対象ファイルの自動検知だ。AIエージェントが編集するファイルは固定じゃない。`watchexec`が変更を検知したファイルをそのまま`glow`に渡す方式が最も自然だ：

```bash
watchexec -w . -e md --ignore node_modules --ignore .git -r -- sh -c '
  FILE="${WATCHEXEC_WRITTEN_PATH:-}"
  if [ -z "$FILE" ] || [ ! -f "$FILE" ]; then
    FILE=$(find . -name "*.md" -maxdepth 3 -exec stat -f "%m %N" {} \; | sort -rn | head -1 | cut -d" " -f2-)
  fi
  [ -n "$FILE" ] && clear && glow -p "$FILE"
'
```

**灰原**: `find`の`-printf`はGNU findの機能でmacOSでは使えない。`stat -f`を使っているのは正しいが、大規模プロジェクトでfindを毎回走らせるのはパフォーマンスの問題がある。

**九条**: `WATCHEXEC_WRITTEN_PATH`がある場合はfindを走らせない。フォールバックとしてのみfindを使う。初回起動時と、環境変数が未設定の場合だけだ。実用上は問題にならない。

**灰原**: ……なら許容範囲だ。ただし`--ignore`に`build`、`dist`、`.next`、`target`、`vendor`も追加すべきだ。

**九条**: 同意する。これをラッパースクリプトにまとめて、`cmux.json`からはスクリプトを呼ぶだけにしよう。設定の実体をスクリプトに分離すれば、cmux.jsonを編集せずにプレビューの挙動を調整できる。

## 第3部: レイアウト設計 ── 最適なペイン配置を決める

**風間**: さて、本丸だ。ペインのレイアウトをどうするか。ベストを考えよう。

**九条**: レイアウト設計の原則から始めよう。3つある。

**第一原則: 面積は注意の優先度に比例させる。** AIエージェントが最も長時間見るペインだから、最大面積を割り当てる。60-75%は必要だ。

**第二原則: 関連情報は視線移動を最小化する位置に置く。** AIの出力を見ながら、ファイル構造やgit diffを参照する。左にサブ情報、右にメイン情報という配置が、左→右の自然な視線移動に合致する。

**第三原則: 頻度の低い操作はタブ（surface）に隠す。** 常時表示が不要なものは、同一ペイン内のタブ切り替えで対応する。cmuxのsurface機能がこれを可能にする。

**桜井**: タブ切り替えって、ペインの中にさらにタブがあるってことですか？

**九条**: そうだ。cmuxの構造はWindow > Workspace > Pane > Surfaceという4層になっている。一つのペインの中に複数のsurface（タブ）を持てる。`Cmd+[`と`Cmd+]`で切り替える。これを使えば、物理的なペイン数を増やさずに機能を追加できる。

### 案の提示と批判

**風間**: じゃあ僕の提案。3カラムにしよう！

```
┌────────┬───────────────┬────────────┐
│        │               │  markdown  │
│  yazi  │  Claude Code  │  preview   │
│        │    (メイン)    ├────────────┤
├────────┤               │   shell    │
│lazygit │               │            │
└────────┴───────────────┴────────────┘
```

左カラムにyaziとlazygit、中央にAIターミナル、右にマークダウンプレビューとshell。情報がきれいに分かれる！

**灰原**: 3カラムは一見整理されて見えるが、問題がある。一般的なモニター幅が2560pxだとして、3等分すると各カラム約853px。中央のAIターミナルに50%割り当てても1280px、左右が各640px。AIエージェントのテキスト出力は横幅が命だ。1280pxでは80カラムがぎりぎりで、インデントの深いコードや長いエラーメッセージが折り返される。

**九条**: 灰原の計算は正確だ。さらに根本的な問題がある。3カラムは**人間の視線の移動コストが高い**。左端のyaziを見て、中央のAIターミナルを見て、右端のマークダウンを見る。視線が画面を横断する。2カラムなら左右だけで済む。

**風間**: うーん、確かに視線がバタバタするか……。じゃあ参考レイアウトに近い形はどうだ？

```
┌──────────┬─────────────────────┐
│          │   shell (小さめ)     │
│   yazi   ├─────────────────────┤
│          │                     │
├──────────┤    AI Terminal      │
│ lazygit  │      (メイン)        │
└──────────┴─────────────────────┘
```

**灰原**: これだとマークダウンプレビューの居場所がない。yaziペインにMDプレビューをタブで入れるとして、shellが右上にある意味は何だ？ shellはメインの作業を補助するものだから、AIターミナルの近くに置いた方が視線移動が少ない。

**九条**: 灰原に同意する。shellは右下に置くべきだ。右上の小さなペインにshellを入れても、表示面積が小さすぎて使いにくい。それよりもAIターミナルを右カラムの上部で最大化して、shellは下部に薄く置く方が合理的だ。

### 最終案への収束

**九条**: 2カラム構成を提案する。左カラムにサブツール群、右カラムにAIターミナル + shell。そして左カラムの上段にyaziとマークダウンプレビューをsurface（タブ）で同居させる。

```
┌─────────────────┬────────────────────────────────────┐
│ [yazi | MD Prev] │                                    │
│  surfaces切替    │       AI Terminal (メイン)           │
│  (左25%, 上60%)  │        (右75%, 上80%)               │
├─────────────────┤                                    │
│                 ├────────────────────────────────────┤
│    lazygit      │             Shell                  │
│  (左25%, 下40%) │          (右75%, 下20%)             │
└─────────────────┴────────────────────────────────────┘
```

**桜井**: あ、これだとyaziを見たいときはyaziのタブ、マークダウンを確認したいときはMDのタブに切り替えるんですね。同時には見られないけど、両方常に表示する必要もないから合理的ですね。

**九条**: その通り。yaziとマークダウンプレビューは「同時に見る必要がない」ことが鍵だ。ファイル構造を確認するフェーズと、出力ドキュメントを確認するフェーズは時間的に重ならない。ならばタブで切り替える方が、各ツールに使える面積を最大化できる。

**灰原**: サイズ比率はどうする。

**九条**: 左カラム25%、右カラム75%。左カラム内は上段60%、下段40%。右カラム内は上段80%、下段20%。

**風間**: 右下のshellが20%って小さすぎない？

**九条**: shellは補助用途だ。`npm install`、`brew install`、ちょっとしたファイル操作。AIエージェントが使えない局面で一時的に使うだけだ。大きな作業はAIエージェントに任せる。20%で十分だ。

**桜井**: 各ペインのサイズは後から変えられるんですか？

**九条**: cmuxではペインの境界をドラッグしてリサイズできる。`cmux.json`の`size`はあくまで初期比率だ。作業中に「lazygitをもっと大きく見たい」と思ったら、境界をドラッグすればいい。

## 第4部: メインターミナルの設計思想 ── なぜzshなのか

**風間**: ここで大事な設計判断がある。メインペインのcommandを何にするかだ。最初は`claude`を直接指定しようとしたけど、考え直した方がいい。

**桜井**: `claude`を直接起動しちゃだめなんですか？

**九条**: ダメではないが、制約が生まれる。`command: "claude"`にすると、そのペインはClaude Code専用になる。Claude Codeを終了したらペインのプロセスが終了して、ペイン自体が閉じる可能性がある。さらに重要なのは、**Claude Codeだけがこの世のAIエージェントではない**ということだ。

**風間**: そう！ Gemini CodeやCodex CLIも出てきている。メインペインを特定のツールに縛ると、ツールを切り替えるたびにcmux.jsonを書き換えることになる。

**九条**: 正解は`command: "zsh"`だ。素のシェルを起動しておけば、ユーザーが宣言的にコマンドを打てる。

```bash
claude              # Claude Code
claude --resume     # 前回セッション再開
claude --model opus # モデル指定
gemini              # Gemini
codex               # Codex
```

セッションが終わってもシェルは生き続ける。別のエージェントに切り替えるのもコマンド一つだ。

**灰原**: これはcmux.jsonの保守性にも影響する。`command: "zsh"`なら設定ファイルを一度書いたら二度と触らなくていい。ツール固有のオプションはユーザーの手元で制御できる。

**桜井**: あ、そうか。cmux.jsonは「部屋の間取り」を定義するもので、「部屋で何をするか」は住む人が決めるんですね。

**九条**: 完璧な比喩だ。cmux.jsonはワークスペースのトポロジーを定義する。その上で何を実行するかは、ユーザーのコンテキストに委ねる。これが**ツール非依存**の設計だ。

## 第5部: 設定の一元管理 ── dotfilesパターン

**灰原**: ここで運用の話をしたい。cmux.json、yaziの設定、glowのラッパースクリプト、Ghosttyの設定変更。設定ファイルが複数の場所に散らばっている。これは管理上の問題だ。

**九条**: dotfilesパターンで解決する。設定ファイルの実体を一つのGitリポジトリに集約し、実際の配置場所にはシンボリックリンクを張る。

**桜井**: dotfilesパターンって何ですか？

**九条**: Unixの慣習で、ホームディレクトリの設定ファイル（`.bashrc`、`.gitconfig`など）をGitリポジトリで管理する手法だ。リポジトリをクローンしてセットアップスクリプトを実行すれば、どのマシンでも同じ開発環境を再現できる。

**風間**: つまりこういう構成だな：

```
cmux-workspace/                  <- GitHubリポジトリ
├── setup.sh                     # deps install + symlink
├── cmux.json                    # ワークスペースレイアウト
├── scripts/
│   └── md-preview.sh            # MD preview wrapper
├── config/
│   ├── yazi.toml                # yazi設定
│   └── ghostty-append.conf      # ghostty追加設定
├── docs/
│   └── meeting.md               # この会議の議事録
├── .gitignore
└── README.md
```

**灰原**: `setup.sh`は何をする？

**九条**: 3つのフェーズだ。第一に、依存ツールのインストール。Homebrewでcmux、yazi、lazygit、glow、watchexecをインストールする。Claude Codeも`brew install --cask claude-code`で入る。第二に、設定ファイルのシンボリックリンク作成。`config/yazi.toml`を`~/.config/yazi/yazi.toml`にリンクする。スクリプトも同様だ。第三に、Ghosttyの設定にscrollback-limitを追記する。

**灰原**: 既存の設定ファイルがある場合は？

**九条**: バックアップを作ってからリンクする。既存ファイルが見つかったら`.bak`にリネームしてからシンボリックリンクを張る。

**風間**: チームメンバーが新しくjoinしたら？

**九条**: `git clone` + `./setup.sh`で完了だ。プロジェクトで使うときは、プロジェクトルートにcmux.jsonをシンボリックリンクする：

```bash
ln -sf ~/app/cmux-workspace/cmux.json /path/to/project/cmux.json
```

**桜井**: 設定を変えたいときは？

**九条**: リポジトリ内のファイルを編集してgit commitする。シンボリックリンクだから、リポジトリ内のファイルを変えれば実際の設定も即座に反映される。`git diff`で変更履歴を追えるし、`git revert`で戻せる。

**灰原**: これなら設定の変更管理が完全にGitのワークフローに乗る。レビューも可能だ。

## 第6部: エッジケースと運用上の注意

**灰原**: 最後に、運用上のリスクを洗い出したい。

### 懸念1: watchexecのCPU消費

**灰原**: 大規模リポジトリで`-w .`するとFSEventsの監視対象が膨大になる。`node_modules`、`.git`、`build`ディレクトリを除外すべきだ。

**九条**: `watchexec`は`--ignore`オプションでパターン除外できる。md-preview.shではすでに主要なディレクトリを除外している。プロジェクト固有の大規模ディレクトリ（`.next`、`target`、`vendor`など）があれば、スクリプトの`IGNORE_DIRS`変数に追加すればいい。

### 懸念2: lazygitとAIエージェントの競合

**灰原**: AIエージェントがgit操作（commit, branch作成）を行った瞬間、lazygitの表示が古くなる。自動更新にラグがある。

**九条**: lazygitはgitディレクトリのFSEventsを監視していて、通常は1-2秒以内に更新される。気になるなら、lazygit側で`r`キーを押してリフレッシュを手動トリガーすればいい。

### 懸念3: ディスプレイサイズの変化

**灰原**: 外部モニター → ラップトップ画面に変えたとき、ペインの比率が崩れないか。

**九条**: cmux.jsonの`size`は比率指定（0.0〜1.0）だから、ウィンドウサイズに応じて自動スケーリングされる。ただし13インチのラップトップだと左カラム25%＝約400px。yaziのファイル名が途切れる可能性はある。

**風間**: そのときはyaziのファイル名表示を短縮する設定があるはずだ。

**九条**: 設定済みだ。`config/yazi.toml`で`linemode = "none"`にしてある。ファイルサイズやタイムスタンプを非表示にして、ファイル名に使える幅を最大化している。

### 懸念4: cmuxアップデートによる設定互換性

**灰原**: cmuxは活発に開発されているプロジェクトだ。cmux.jsonのスキーマが変わったらどうする？

**九条**: それは現実的なリスクだ。対策は2つ。第一に、cmux.jsonのスキーマはcmuxのバージョンとともにドキュメントされている。`setup.sh`にcmuxのバージョンチェックを入れて、互換性のあるバージョンを明示する。第二に、GitHubリポジトリでIssueやリリースノートを追跡して、破壊的変更に早期に対応する。

**桜井**: cmux自体がHomebrew経由でインストールされるから、`brew upgrade`で更新されちゃいますよね？

**灰原**: その通りだ。意図しないアップグレードを防ぐなら`brew pin cmux`でバージョンを固定できる。ただし、セキュリティパッチも止まるから推奨はしない。

**九条**: 現実的には、cmux.jsonのスキーマは基本的な`split`/`pane`/`surfaces`の構造が大きく変わることは考えにくい。仮に変わった場合でも、修正はJSONの数行で済む。Gitで管理していれば差分も明確だ。

## 議事録 ── 天才ブロガー視点の構造化まとめ

> **筆者: 九条 玲**
> *ターミナルは思考の延長であるべきだ。今回の会議で、AIコーディングエージェントのための最適なcmuxワークスペースを設計した。以下に、議論の要点と最終成果物を構造化して記録する。*

---

### 1. 前提: なぜcmuxなのか

| 比較軸 | tmux + tmuxinator | cmux |
|--------|-------------------|------|
| 設定方式 | YAML + シェルスクリプト | JSON（宣言的） |
| AI統合 | 手動フック構築が必要 | ネイティブ（通知、チームモード） |
| レンダリング | ホストターミナルに依存 | GPU加速（libghostty） |
| ブラウザ統合 | なし | WebKit内蔵ペイン |
| スクリプトAPI | tmuxコマンド | CLI + ソケットAPI |
| セッション復元 | tmux-resurrect | ネイティブ（レイアウト+scrollback） |
| 対応OS | 全OS | macOS 14+のみ |

**結論**: AIエージェントをメインで使うmacOS環境では、cmuxが統合性・設定容易性・パフォーマンスのすべてにおいて優位。

---

### 2. ツール選定の根拠

| ペイン | ツール | 選定理由 | 却下した代替案と理由 |
|--------|--------|----------|---------------------|
| ファイルツリー | **yazi** | 非同期I/O、起動高速、ratio設定で狭ペイン対応 | ranger（Python、起動遅い）、nnn（プラグイン機構弱い） |
| Git | **lazygit** | diff+staging+commit統合TUI | `watch git diff`（情報密度低い、操作不可） |
| MDプレビュー | **glow + watchexec** | 軽量、ターミナル完結、pager無しでwatch連携 | grip+ブラウザペイン（Python依存、WebKitプロセス追加、API制限） |
| メイン | **zsh** | ツール非依存。claude/gemini/codexを自由に起動 | claude直接起動（ツールロックイン、終了時ペイン消失） |
| 補助 | **zsh** | 汎用シェル | - |

---

### 3. 最終レイアウト

```
┌─────────────────┬────────────────────────────────────┐
│ [yazi | MD Prev] │                                    │
│  surfaces切替    │       AI Terminal (メイン)           │
│  (左25%, 上60%)  │        (右75%, 上80%)               │
├─────────────────┤                                    │
│                 ├────────────────────────────────────┤
│    lazygit      │             Shell                  │
│  (左25%, 下40%) │          (右75%, 下20%)             │
└─────────────────┴────────────────────────────────────┘
```

**設計思想**:
- AIターミナルに全体面積の60%（75% x 80%）を確保
- yazi/MDプレビューはsurface切替で同居（同時閲覧不要の原則）
- 左カラム = 参照情報、右カラム = 操作対象（視線移動の最小化）
- shellは最小限（AIエージェントに委譲できない操作の受け皿）
- メインペインはzsh（ツール非依存、AIエージェント切替自由）

---

### 4. 設計原則のまとめ

1. **面積は注意の優先度に比例させる** — 最も注視するペインに最大面積
2. **視線移動を最小化する** — 2カラム構成、左=参照/右=操作
3. **頻度の低い操作はタブに隠す** — surfaces機能で物理ペイン数を抑制
4. **目的から逆算して選ぶ** — 「使えるから使う」は設計ではない
5. **ツール非依存** — cmux.jsonは部屋の間取り、何をするかは住む人が決める
6. **設定はGitで管理する** — dotfilesパターンで再現性と追跡可能性を確保
